关键洞察

Webpack 和 Rollup 是 JS 时代的开拓者，分别定义了"应用打包"和"库打包"两条路线。Rspack 和 Rolldown 分别是它们的 Rust 重写版，API 兼容但速度提升数十倍。Vite 站在中间做整合——开发用 ESM 原生加载，生产用 Rolldown（从 v8 开始）打包。esbuild 是底层基础设施，被 Vite、tsup、tsx 等大量工具依赖。Turbopack 目前与 Next.js 强绑定，无法独立使用。

关键洞察

Vite 的"开发 Unbundled + 生产 Bundle"双架构是一把双刃剑：开发体验极好（毫秒级启动），但开发和生产行为不完全一致可能导致"开发没问题、上线才暴露"的情况。Webpack/Rspack 的统一架构虽然开发启动慢一些，但行为一致性更强。Turbopack 的增量计算引擎是最激进的设计——只重新构建真正变化的部分，理论上 HMR 速度不随项目规模增长。

关键洞察

现代构建工具已经彻底放弃 tsc 做构建转译。esbuild / SWC / oxc 只做语法转换不做类型检查，速度快几十倍。最佳实践是：构建时用快速转译器，类型检查放在 IDE 实时检查 + CI 流水线的 tsc --noEmit 中。CSS 处理是 Webpack 配置复杂度的重灾区——同一个"导入 CSS"的需求，Webpack 需要四层 loader 配置，Vite 零配置开箱即用。

Rollup 的产物是所有工具中最干净的——输出代码最接近手写 ESM 模块，几乎无额外运行时。这是它作为库打包工具的核心优势。压缩器的选择体现速度和压缩率的权衡：Terser 压缩率最高但最慢，esbuild 最快但略低，SWC 是折中方案（接近 Terser 的效果但快得多）。

从 Webpack 迁移的难度

迁移到难度说明
Rspack🟢 低API 几乎 100% 兼容，大部分项目改包名即可，是最低成本方案
Vite🟡 中配置需完全重写（loader → plugin），但源代码基本不改
Rollup🔴 高需重新设计构建流程，处理 HTML、dev server 等，不推荐应用项目迁移
Rolldown🟡 中兼容 Rollup 配置，但仍处早期，直接用不如通过 Vite 间接使用
esbuild🔴 高功能覆盖面窄，缺 CSS Modules、HTML 处理等，不适合替代 Webpack
Turbopack🔴 高必须迁移到 Next.js 框架，是整个技术栈的迁移而非构建工具替换

新项目选型建议

通用 Web 应用（React / Vue / Svelte）

推荐 Vite

开发体验最好，生态丰富，社区最活跃，是当下最佳平衡点

大型 Webpack 存量项目提速

推荐 Rspack

API 兼容 Webpack，迁移成本最低，构建速度立竿见影提升 5-10 倍

打包 npm 库 / 框架

推荐 Rollup 或 Vite lib mode

产物最干净，Tree Shaking 最强，React/Vue/Svelte 自身都用它构建

Next.js 应用

推荐 Turbopack

Next.js 原生集成，开箱即用，增量构建体验优秀

需要极致构建速度的 CI/CD

推荐 esbuild（简单场景）/ Rolldown（复杂场景）

速度优先，适合对构建时间敏感的持续集成流水线

新项目且不确定技术栈

推荐 Vite

社区趋势、生态丰富度、性能表现的最佳平衡点，未来可平滑演进

总结

前端构建工具的竞争本质上是"用什么语言写构建工具"的竞争。Webpack 和 Rollup 作为 JS 实现的开拓者定义了规则，而 Rspack 和 Rolldown 作为 Rust 重写版正在继承规则的同时打破速度瓶颈。Vite 则站在中间，用巧妙的架构设计在当下取得了最好的平衡。选择工具时，不要只看速度——配置复杂度、插件生态、迁移成本和团队熟悉度同样重要。
